## 重要な制約: 分割の単位

分割の単位は「機能単位」です。以下の原則に従ってください。

### ✅ 正しい分割（機能単位）
各子Issueが独立した機能として完結し、単独でPRを作成・レビュー・マージできること。
例:
- 子Issue A: CLIオプションのパースとバリデーション機能
- 子Issue B: エージェント連携とJSONパース機能
- 子Issue C: GitHub API操作（Issue作成・コメント投稿）機能

### ❌ 禁止する分割（工程単位）
「要件定義」「実装」「テスト」のような作業工程での分割は行わないこと。
テストは各機能Issueの完了条件として本文中に記載すること。
例（これは禁止）:
- 子Issue 1: ○○の要件定義
- 子Issue 2: ○○の実装
- 子Issue 3: ○○のテスト

## 出力形式

以下のJSON形式で**指定されたファイルパスに書き込んで**ください：

**出力先ファイルパス**: `{OUTPUT_FILE_PATH}`

```json
{
  "summary": "分割の概要説明（1〜3文）",
  "issues": [
    {
      "title": "子 Issue のタイトル（80 文字以内）",
      "body": "子 Issue の本文（Markdown 形式）",
      "labels": ["enhancement"],
      "priority": "high | medium | low",
      "relatedFeatures": ["関連する他の子 Issue のタイトル"]
    }
  ],
  "metrics": {
    "completeness": 85,
    "specificity": 70
  }
}
```

**重要**: 必ず上記の絶対パスにJSONファイルをWriteツールで書き込んでください。コンソールへの出力ではなく、ファイルとして保存することが必須です。相対パスや `/workspace` プレフィックスは使用しないでください。

## 品質メトリクス
- completeness: 必要なセクションの網羅率（0-100）
- specificity: 具体的な情報の含有率（0-100）
