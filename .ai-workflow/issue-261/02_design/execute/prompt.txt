### 本文

## 概要

ワークフロー完了時の最終処理を行う `finalize` コマンドを新規追加する。

## 背景

現在、ワークフロー完了後の後処理（クリーンアップ、コミットスカッシュ、PR更新、ドラフト解除）は個別に実行する必要がある。これらを統合した `finalize` コマンドを作成することで、ワークフロー完了時の作業を自動化・簡素化できる。

また、`execute` コマンドの `--squash-on-complete` オプションは finalize コマンドに責務を移すため、Job DSL のデフォルト値を `false` に変更する。

## finalize コマンドの処理ステップ

| ステップ | 内容 | 詳細 |
|---------|------|------|
| 1 | base_commit 一時保存 | metadata.json の base_commit を取得・メモリ保持 |
| 2 | .ai-workflow フォルダ削除 + コミット | ワークフローディレクトリ全体を削除してコミット |
| 3 | コミットスカッシュ | base_commit から HEAD までをスカッシュ |
| 4 | PR 内容の最終更新 | PR 本文を最終版に更新 |
| 5 | PR ドラフト解除 | Draft → Ready for Review に変更 |

### 2. `PullRequestClient.markPRReady()` - PR ドラフト解除

```typescript
// GitHub GraphQL API または gh コマンドで実装
public async markPRReady(prNumber: number): Promise<GenericResult>
```

GitHub GraphQL API の mutation:
```graphql
mutation MarkPRReady($pullRequestId: ID!) {
  markPullRequestReadyForReview(input: { pullRequestId: $pullRequestId }) {
    pullRequest {
      isDraft
    }
  }
}
```

または `gh pr ready <PR_NUMBER>` コマンドを subprocess で実行。

### 3. `PullRequestClient.updateBaseBranch()` - PR マージ先ブランチ変更

```typescript
// PR のマージ先ブランチを変更
public async updateBaseBranch(prNumber: number, baseBranch: string): Promise<GenericResult>
```

GitHub REST API:
```
PATCH /repos/{owner}/{repo}/pulls/{pull_number}
{
  "base": "develop"
}
```

### 4. PR 最終本文生成ロジック

変更サマリー、完了ステータス、テスト結果などを含む PR 本文を生成。

## CLI オプション

```bash
# 基本的な使用方法
node dist/index.js finalize --issue 123

# ドライラン（実際には実行しない）
node dist/index.js finalize --issue 123 --dry-run

# スカッシュをスキップ
node dist/index.js finalize --issue 123 --skip-squash

# PR 更新をスキップ
node dist/index.js finalize --issue 123 --skip-pr-update

# マージ先ブランチを指定（デフォルト: main）
node dist/index.js finalize --issue 123 --base-branch develop
```

## Job DSL の変更

### 1. `jenkins/jobs/dsl/ai-workflow/ai_workflow_all_phases_job.groovy`

`SQUASH_ON_COMPLETE` パラメータのデフォルト値を `true` → `false` に変更:

```groovy
// Before
booleanParam('SQUASH_ON_COMPLETE', true, ...)

// After
booleanParam('SQUASH_ON_COMPLETE', false, ...)
```

### 2. `jenkins/jobs/dsl/ai-workflow/ai_workflow_preset_job.groovy`

`SQUASH_ON_COMPLETE` パラメータのデフォルト値を `true` → `false` に変更:

```groovy
// Before
booleanParam('SQUASH_ON_COMPLETE', true, ...)

// After
booleanParam('SQUASH_ON_COMPLETE', false, ...)
```

**理由**: スカッシュ処理は finalize コマンドの責務に移行するため、execute コマンドでのデフォルト動作は無効化する。

## 実装の流れ

```
finalize コマンド
    │
    ├─► Step 1: base_commit 取得・一時保存
    │     └─ MetadataManager.getBaseCommit() ← 既存
    │
    ├─► Step 2: .ai-workflow 削除 + コミット
    │     └─ ArtifactCleaner.cleanupWorkflowArtifacts() ← 既存
    │         └─ GitManager.commitCleanupLogs() ← 既存
    │
    ├─► Step 3: コミットスカッシュ
    │     └─ SquashManager（修正版）
    │         └─ RemoteManager.forcePushToRemote() ← 既存
    │
    ├─► Step 4: PR 本文更新 + マージ先変更
    │     ├─ PullRequestClient.updatePullRequest() ← 既存
    │     ├─ PullRequestClient.updateBaseBranch() ← 新規追加
    │     └─ 新規: PR最終本文生成ロジック
    │
    └─► Step 5: PR ドラフト解除
          └─ PullRequestClient.markPRReady() ← 新規追加
```

## 課題と対応

| 課題 | 対応案 |
|-----|-------|
| SquashManager は PhaseContext を必要とする | finalize 用のシンプルなコンテキストを作成、またはフォールバックメッセージのみ使用 |
| ドラフト解除は GraphQL API が必要 | `gh pr ready` コマンドを subprocess で実行するか、GraphQL mutation を実装 |
| Step 2 の削除後に base_commit が消える | Step 1 でメモリに保持し、Step 3 で使用 |
| PR 番号の取得 | metadata.json の `pr_number` または `PullRequestClient.checkExistingPr()` |
| マージ先ブランチの変更 | REST API の PATCH で `base` フィールドを更新 |

## 設計書作成手順

### 1. 既存コードベース分析

要件定義書とIssue内容から、以下を分析してください：

1. **影響範囲の特定**
   - どのコンポーネント（Ansible/Pulumi/Jenkins/Scripts）が影響を受けるか
   - 既存ファイルの修正が必要か、新規ファイルの作成が必要か
   - 依存関係のあるコンポーネントは何か

2. **既存パターンの調査**
   - 類似の機能実装がないか調査
   - 既存のコーディング規約やパターンを確認
   - 再利用可能なコンポーネントの特定

### 2. 実装戦略の判断（必須）

以下の3つから選択し、**判断根拠を明記**してください：

- **CREATE（新規作成）**: 既存コードとの依存関係がなく、完全に新しい機能
- **EXTEND（拡張）**: 既存コードを拡張・修正する必要がある
- **REFACTOR（リファクタリング）**: 既存コードの構造改善が主目的

**判断基準**：
- 既存ファイルへの影響範囲
- 新規ファイルの作成数
- 既存機能との統合度

**出力形式**：
```markdown
### 実装戦略: CREATE / EXTEND / REFACTOR

**判断根拠**:
- （理由1）
- （理由2）
```

### 3. テスト戦略の判断（必須）

以下から選択し、**判断根拠を明記**してください：

- **UNIT_ONLY**: Unitテストのみ
- **INTEGRATION_ONLY**: Integrationテストのみ
- **BDD_ONLY**: BDDテストのみ
- **UNIT_INTEGRATION**: Unit + Integration
- **UNIT_BDD**: Unit + BDD
- **INTEGRATION_BDD**: Integration + BDD
- **ALL**: すべて

**判断基準**：
- 機能の複雑度
- ユーザーストーリーの有無
- 既存テストとの整合性

**出力形式**：
```markdown
### テスト戦略: UNIT_ONLY / INTEGRATION_ONLY / BDD_ONLY / UNIT_INTEGRATION / UNIT_BDD / INTEGRATION_BDD / ALL

**判断根拠**:
- （理由1）
- （理由2）
```

### 4. テストコード戦略の判断（必須）

以下から選択し、**判断根拠を明記**してください：

- **EXTEND_TEST（既存テストの拡張）**: 既存テストファイルに追加
- **CREATE_TEST（新規テスト作成）**: 新しいテストファイルを作成
- **BOTH_TEST（両方）**: 既存テストの拡張と新規テストの作成

**判断基準**：
- 既存テストファイルの有無
- 既存テストとの関連性

**出力形式**：
```markdown
### テストコード戦略: EXTEND_TEST / CREATE_TEST / BOTH_TEST

**判断根拠**:
- （理由1）
- （理由2）
```

### 5. 設計書の構成

以下のセクションを含む設計書を作成してください：

#### 必須セクション：

1. **アーキテクチャ設計**
   - システム全体図
   - コンポーネント間の関係
   - データフロー

2. **実装戦略判断**（セクション2で記載）

3. **テスト戦略判断**（セクション3で記載）

4. **テストコード戦略判断**（セクション4で記載）

5. **影響範囲分析**
   - 既存コードへの影響
   - 依存関係の変更
   - マイグレーション要否

6. **変更・追加ファイルリスト**
   - 新規作成ファイル（相対パス）
   - 修正が必要な既存ファイル（相対パス）
   - 削除が必要なファイル（相対パス）

7. **詳細設計**
   - クラス設計（必要な場合）
   - 関数設計（主要な関数のみ）
   - データ構造設計
   - インターフェース設計

8. **セキュリティ考慮事項**
   - 認証・認可
   - データ保護
   - セキュリティリスクと対策

9. **非機能要件への対応**
   - パフォーマンス
   - スケーラビリティ
   - 保守性

10. **実装の順序**
    - 実装順序の推奨
    - 依存関係の考慮
