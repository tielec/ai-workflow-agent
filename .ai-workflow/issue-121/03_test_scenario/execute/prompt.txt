### 本文

## 概要

AI Workflow Agentにリポジトリを探索して自動的にIssueを作成する機能を追加する。現在はIssueを手動で作成してからワークフローを実行しているが、Issue作成自体もAIエージェントに任せることで、開発効率をさらに向上させる。

## 背景・動機

- Issue作成作業が負担になってきた
- リポジトリ全体を探索して潜在的な問題や改善点を発見したい
- 既存のIssueと重複しないように自動的にチェックしたい

## 実装すべき機能

### 1. リポジトリ探索エンジン

コードベース全体を探索し、以下の3つのカテゴリで問題・改善点を検出する：

#### a. 潜在的な不具合の報告
- エラーハンドリングの欠如
- 型安全性の問題（`any`型の過剰使用等）
- リソースリーク（unclosed streams, event listeners等）
- セキュリティ上の懸念（credentials漏洩、XSS脆弱性等）
- 依存関係の脆弱性

#### b. リファクタリングすべきコードやドキュメントの報告
- 大きすぎるファイル・関数（行数基準）
- 重複コード
- 複雑度が高いコード（Cyclomatic Complexity）
- 未使用のインポート・変数
- ドキュメントの欠落または古い記述

#### c. 将来的な機能拡張のアイデアの提案

**既存機能の改善**:
- ユーザビリティ改善の提案
- パフォーマンス最適化の提案
- テストカバレッジ向上の提案
- CI/CD改善の提案

**創造的な新機能の提案**（柔軟な発想で）:
- **他ツール・サービスとの連携アイデア**
  - Slack/Discord通知、Jiraチケット自動生成、VSCode拡張機能等
- **ワークフロー自動化の拡張**
  - 定期実行による技術的負債の可視化、依存関係自動更新等
- **開発者体験（DX）向上**
  - インタラクティブCLI、進捗ダッシュボード、AIペアプログラミング支援等
- **品質保証の強化**
  - セキュリティスキャン統合、アクセシビリティチェック、負荷テスト自動化等
- **データ駆動の意思決定支援**
  - メトリクスダッシュボード、コスト分析レポート、開発速度予測等
- **エコシステム拡張**
  - プラグインシステム、カスタムフェーズ定義、テンプレート共有マーケットプレイス等
- **その他、リポジトリ内容から着想を得た独自のアイデア**

AIエージェントには、既存の枠にとらわれず、リポジトリの特性やコミット履歴、ドキュメント内容から「こういう機能があったらもっと便利になる」という提案を自由に発想してほしい。

### 2. 重複Issue検出機能

- GitHub APIを使用して既存のIssue一覧を取得
- LLM（OpenAI APIまたはAnthropic API）を使用して意味的類似度を判定
- 重複と判定されたIssueは作成をスキップし、ログに記録

### 3. Issue自動作成フロー

```bash
# 新しいCLIコマンド
node dist/index.js auto-issue [OPTIONS]

# オプション例
--category <bug|refactor|enhancement|all>  # 作成するIssueのカテゴリ
--limit <NUM>                              # 作成するIssueの最大数（デフォルト: 5）
--dry-run                                  # 実際には作成せず、候補のみ表示
--similarity-threshold <0-1>               # 重複判定の類似度閾値（デフォルト: 0.8）
--creative-mode                            # 創造的な機能提案を優先的に生成
```

### 4. Issue作成テンプレート

生成されるIssueは以下の形式に従う：

```markdown
## 概要
[AIが生成した問題・改善点のサマリー]

## 詳細
[具体的な問題の説明]

## 該当箇所
- ファイル: src/path/to/file.ts:123-145
- 関連コード: [コードスニペット]

## 提案される解決策
[AIによる解決策の提案]

## 期待される効果
[この改善による具体的なメリット]

## 優先度
[Low / Medium / High]

## カテゴリ
[bug / refactor / enhancement]

```

## 技術的な実装アプローチ

### 1. 新しいコマンドハンドラ
- `src/commands/auto-issue.ts`: `handleAutoIssueCommand()`

### 2. リポジトリ探索エンジン
- `src/core/repository-analyzer.ts`: コードベース解析
  - 静的解析（TypeScript AST解析）
  - パターンマッチング（ESLintルール活用）
  - メトリクス計測（複雑度、行数等）
  - コミット履歴分析（頻繁に変更されるファイル、バグ修正パターン等）
  - ドキュメント分析（README、CLAUDE.md、ARCHITECTURE.md等）

### 3. 重複検出エンジン
- `src/core/issue-deduplicator.ts`: 既存Issue比較
  - GitHub API連携（既存Issue取得）
  - LLM活用（意味的類似度判定）
  - 類似度スコアリング

### 4. Issue生成エンジン
- `src/core/issue-generator.ts`: Issue自動作成
  - テンプレート生成
  - LLMによる本文生成（創造的な提案を引き出すプロンプト設計）
  - GitHub API経由でIssue作成

### 5. 創造的提案のためのLLMプロンプト設計
- リポジトリの特性を分析し、文脈に応じた提案を生成
- 既存機能の組み合わせや拡張による新しいユースケースの発見
- 他のプロジェクトやツールからのベストプラクティス適用

### 6. 既存モジュールの活用
- `src/core/github-client.ts`: GitHub API連携（Issue作成・取得）
- `src/core/config.ts`: 環境変数管理
- `src/utils/logger.ts`: ロギング

## 期待される効果

1. **開発効率の向上**: Issue作成作業の自動化
2. **品質向上**: 見落としがちな潜在的問題の早期発見
3. **継続的改善**: 定期的な自動Issue作成による技術的負債の可視化
4. **重複削減**: 既存Issueとの重複を自動的に回避
5. **イノベーション促進**: AIによる創造的な機能提案により、新しい可能性を発見

## 代替案・考慮事項

### 代替案1: 完全自動実行（Cron Job）
- 週次で自動的にリポジトリを探索し、Issueを作成
- GitHub Actionsとの連携

### 代替案2: 段階的実装
- Phase 1: バグ検出のみ実装
- Phase 2: リファクタリング検出を追加
- Phase 3: 機能拡張提案を追加（創造的提案を含む）

### 考慮事項
- **LLMコスト**: 大規模リポジトリでのトークン使用量
- **誤検知**: AIによる誤ったIssue作成の可能性
- **Issue spam**: あまりに多くのIssueが作成されることを防ぐ制限
- **プライバシー**: プライベートリポジトリのコード内容をLLMに送信することのセキュリティリスク
- **提案品質の担保**: 創造的提案が実際に有用かどうかの判定基準

## テスト戦略別の対応

### UNIT_ONLY: Unitテストのみ
- 各関数・メソッド単位のテストケース
- 正常系・異常系・境界値テスト
- モック/スタブを使用した単体テスト

### INTEGRATION_ONLY: Integrationテストのみ
- コンポーネント間の連携テスト

### BDD_ONLY: BDDテストのみ
- ユーザーストーリーベースのシナリオ
- Given-When-Then形式
- ビジネス要件との対応

## テストシナリオの構成

### 1. テスト戦略サマリー
- 選択されたテスト戦略（Phase 2から引用）
- テスト対象の範囲
- テストの目的

### 2. Unitテストシナリオ（該当する場合）

各関数・メソッドについて：

**テストケース名**: （関数名_正常系/異常系/境界値）

- **目的**: このテストで検証すること
- **前提条件**: テスト実行前の状態
- **入力**: 関数への入力パラメータ
- **期待結果**: 期待される出力・状態変化
- **テストデータ**: 使用するテストデータ

**例**:
```
テストケース名: calculate_total_正常系
目的: 合計金額が正しく計算されることを検証
前提条件: 商品リストが存在する
入力: items = [{price: 100, quantity: 2}, {price: 200, quantity: 1}]
期待結果: 400が返される
テストデータ: 上記items
```

### 3. Integrationテストシナリオ（該当する場合）

各統合ポイントについて：

**シナリオ名**: （統合する2つのコンポーネント名）

- **目的**: この統合で検証すること
- **前提条件**: 統合テスト実行前の状態
- **テスト手順**: ステップバイステップの手順
- **期待結果**: 統合後の期待される動作
- **確認項目**: 確認すべきポイントのチェックリスト

### 4. BDDシナリオ（該当する場合）

ユーザーストーリーごとに：

**Feature**: （機能名）

**Scenario**: （シナリオ名）

- **Given** (前提条件): システムの初期状態
- **When** (操作): ユーザーが実行するアクション
- **Then** (結果): 期待される結果

**例**:
```gherkin
Feature: ユーザー登録機能

Scenario: 新規ユーザーが正常に登録できる
  Given ユーザーがログインしていない
  When ユーザーが有効なメールアドレスとパスワードで登録フォームを送信する
  Then ユーザーアカウントが作成される
  And 確認メールが送信される
  And ユーザーがログイン画面にリダイレクトされる
```

### 5. テストデータ
- 各テストシナリオで使用するテストデータ
- 正常データ、異常データ、境界値データ

### 6. テスト環境要件
- 必要なテスト環境（ローカル、CI/CD等）
- 必要な外部サービス・データベース
- モック/スタブの必要性

## 品質ゲート（Phase 3）

テストシナリオは以下の品質ゲートを満たす必要があります：

- [ ] **Phase 2の戦略に沿ったテストシナリオである**
- [ ] **主要な正常系がカバーされている**
- [ ] **主要な異常系がカバーされている**
- [ ] **期待結果が明確である**

これらの品質ゲートは**必須要件**です。作成後、クリティカルシンキングレビューが実施され、品質ゲートを満たさない場合は修正が必要になります。

## 注意事項

1. **戦略準拠**: Phase 2のテスト戦略を厳守してください
   - 戦略に含まれないテスト種別は作成不要
   - 例: UNIT_ONLYの場合、Integration/BDDシナリオは不要

2. **要件との対応**: 要件定義書の各要件がテストでカバーされていることを確認
   - 受け入れ基準がテストシナリオに反映されているか
   - 機能要件がすべてテスト対象に含まれているか

3. **実行可能性**: 実際に実行可能なテストシナリオを記述
   - 曖昧な表現を避ける
   - 具体的な入力・出力を記載
   - 検証可能な期待結果

4. **優先度**: すべてのエッジケースではなく、主要なケースに注目
   - 80点で十分（完璧を目指さない）
   - クリティカルパス優先
   - 高リスク領域を優先的にカバー

5. **レビューされることを意識**: テストシナリオはクリティカルシンキングレビューが実施されます
   - 品質ゲート（4つの必須要件）を最優先で満たすこと
   - ブロッカー（次フェーズに進めない問題）がないように注意
