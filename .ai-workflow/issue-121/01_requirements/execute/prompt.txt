### 本文

## 概要

AI Workflow Agentにリポジトリを探索して自動的にIssueを作成する機能を追加する。現在はIssueを手動で作成してからワークフローを実行しているが、Issue作成自体もAIエージェントに任せることで、開発効率をさらに向上させる。

## 背景・動機

- Issue作成作業が負担になってきた
- リポジトリ全体を探索して潜在的な問題や改善点を発見したい
- 既存のIssueと重複しないように自動的にチェックしたい

## 実装すべき機能

### 1. リポジトリ探索エンジン

コードベース全体を探索し、以下の3つのカテゴリで問題・改善点を検出する：

#### a. 潜在的な不具合の報告
- エラーハンドリングの欠如
- 型安全性の問題（`any`型の過剰使用等）
- リソースリーク（unclosed streams, event listeners等）
- セキュリティ上の懸念（credentials漏洩、XSS脆弱性等）
- 依存関係の脆弱性

#### b. リファクタリングすべきコードやドキュメントの報告
- 大きすぎるファイル・関数（行数基準）
- 重複コード
- 複雑度が高いコード（Cyclomatic Complexity）
- 未使用のインポート・変数
- ドキュメントの欠落または古い記述

#### c. 将来的な機能拡張のアイデアの提案

**既存機能の改善**:
- ユーザビリティ改善の提案
- パフォーマンス最適化の提案
- テストカバレッジ向上の提案
- CI/CD改善の提案

**創造的な新機能の提案**（柔軟な発想で）:
- **他ツール・サービスとの連携アイデア**
  - Slack/Discord通知、Jiraチケット自動生成、VSCode拡張機能等
- **ワークフロー自動化の拡張**
  - 定期実行による技術的負債の可視化、依存関係自動更新等
- **開発者体験（DX）向上**
  - インタラクティブCLI、進捗ダッシュボード、AIペアプログラミング支援等
- **品質保証の強化**
  - セキュリティスキャン統合、アクセシビリティチェック、負荷テスト自動化等
- **データ駆動の意思決定支援**
  - メトリクスダッシュボード、コスト分析レポート、開発速度予測等
- **エコシステム拡張**
  - プラグインシステム、カスタムフェーズ定義、テンプレート共有マーケットプレイス等
- **その他、リポジトリ内容から着想を得た独自のアイデア**

AIエージェントには、既存の枠にとらわれず、リポジトリの特性やコミット履歴、ドキュメント内容から「こういう機能があったらもっと便利になる」という提案を自由に発想してほしい。

### 2. 重複Issue検出機能

- GitHub APIを使用して既存のIssue一覧を取得
- LLM（OpenAI APIまたはAnthropic API）を使用して意味的類似度を判定
- 重複と判定されたIssueは作成をスキップし、ログに記録

### 3. Issue自動作成フロー

```bash
# 新しいCLIコマンド
node dist/index.js auto-issue [OPTIONS]

# オプション例
--category <bug|refactor|enhancement|all>  # 作成するIssueのカテゴリ
--limit <NUM>                              # 作成するIssueの最大数（デフォルト: 5）
--dry-run                                  # 実際には作成せず、候補のみ表示
--similarity-threshold <0-1>               # 重複判定の類似度閾値（デフォルト: 0.8）
--creative-mode                            # 創造的な機能提案を優先的に生成
```

### 4. Issue作成テンプレート

生成されるIssueは以下の形式に従う：

```markdown
## 概要
[AIが生成した問題・改善点のサマリー]

## 詳細
[具体的な問題の説明]

## 該当箇所
- ファイル: src/path/to/file.ts:123-145
- 関連コード: [コードスニペット]

## 提案される解決策
[AIによる解決策の提案]

## 期待される効果
[この改善による具体的なメリット]

## 優先度
[Low / Medium / High]

## カテゴリ
[bug / refactor / enhancement]

```

## 技術的な実装アプローチ

### 1. 新しいコマンドハンドラ
- `src/commands/auto-issue.ts`: `handleAutoIssueCommand()`

### 2. リポジトリ探索エンジン
- `src/core/repository-analyzer.ts`: コードベース解析
  - 静的解析（TypeScript AST解析）
  - パターンマッチング（ESLintルール活用）
  - メトリクス計測（複雑度、行数等）
  - コミット履歴分析（頻繁に変更されるファイル、バグ修正パターン等）
  - ドキュメント分析（README、CLAUDE.md、ARCHITECTURE.md等）

### 3. 重複検出エンジン
- `src/core/issue-deduplicator.ts`: 既存Issue比較
  - GitHub API連携（既存Issue取得）
  - LLM活用（意味的類似度判定）
  - 類似度スコアリング

### 4. Issue生成エンジン
- `src/core/issue-generator.ts`: Issue自動作成
  - テンプレート生成
  - LLMによる本文生成（創造的な提案を引き出すプロンプト設計）
  - GitHub API経由でIssue作成

### 5. 創造的提案のためのLLMプロンプト設計
- リポジトリの特性を分析し、文脈に応じた提案を生成
- 既存機能の組み合わせや拡張による新しいユースケースの発見
- 他のプロジェクトやツールからのベストプラクティス適用

### 6. 既存モジュールの活用
- `src/core/github-client.ts`: GitHub API連携（Issue作成・取得）
- `src/core/config.ts`: 環境変数管理
- `src/utils/logger.ts`: ロギング

## 期待される効果

1. **開発効率の向上**: Issue作成作業の自動化
2. **品質向上**: 見落としがちな潜在的問題の早期発見
3. **継続的改善**: 定期的な自動Issue作成による技術的負債の可視化
4. **重複削減**: 既存Issueとの重複を自動的に回避
5. **イノベーション促進**: AIによる創造的な機能提案により、新しい可能性を発見

## 代替案・考慮事項

### 代替案1: 完全自動実行（Cron Job）
- 週次で自動的にリポジトリを探索し、Issueを作成
- GitHub Actionsとの連携

### 代替案2: 段階的実装
- Phase 1: バグ検出のみ実装
- Phase 2: リファクタリング検出を追加
- Phase 3: 機能拡張提案を追加（創造的提案を含む）

### 考慮事項
- **LLMコスト**: 大規模リポジトリでのトークン使用量
- **誤検知**: AIによる誤ったIssue作成の可能性
- **Issue spam**: あまりに多くのIssueが作成されることを防ぐ制限
- **プライバシー**: プライベートリポジトリのコード内容をLLMに送信することのセキュリティリスク
- **提案品質の担保**: 創造的提案が実際に有用かどうかの判定基準

## 要件定義書の構成

以下のセクションを含む要件定義書を作成してください：

### 0. Planning Documentの確認（Planning Phaseが実行されている場合）
- 開発計画の全体像を把握
- スコープ、技術選定、リスク、スケジュールを確認
- Planning Documentで策定された戦略を踏まえて要件定義を実施

### 1. 概要
- Issue本文の「## 概要」セクションを要約
- 背景と目的を明確に記述
- ビジネス価値・技術的価値を説明

### 2. 機能要件
- Issue本文の「## TODO」セクションから機能要件を抽出
- 各要件を明確かつ検証可能な形で記述
- 優先度（高/中/低）を付与

### 3. 非機能要件
- パフォーマンス要件
- セキュリティ要件
- 可用性・信頼性要件
- 保守性・拡張性要件

### 4. 制約事項
- 技術的制約（使用技術、既存システムとの整合性）
- リソース制約（時間、人員、予算）
- ポリシー制約（セキュリティポリシー、コーディング規約）

### 5. 前提条件
- システム環境
- 依存コンポーネント
- 外部システム連携

### 6. 受け入れ基準
- 各機能要件の受け入れ基準（Given-When-Then形式推奨）
- テスト可能な形で記述

### 7. スコープ外
- 明確にスコープ外とする事項
- 将来的な拡張候補

## 品質ゲート（Phase 1）

作成する要件定義書は、以下の品質ゲートを満たす必要があります：

- [ ] **機能要件が明確に記載されている**
- [ ] **受け入れ基準が定義されている**
- [ ] **スコープが明確である**
- [ ] **論理的な矛盾がない**

これらの品質ゲートは**必須要件**です。作成後、クリティカルシンキングレビューが実施され、品質ゲートを満たさない場合は修正が必要になります。

## 注意事項

1. **具体性**: 曖昧な表現を避け、具体的かつ測定可能な要件を記述
   - NG例: "適切に処理する"、"必要に応じて対応"
   - OK例: "3秒以内にレスポンスを返す"、"エラー発生時は管理者に通知"

2. **整合性**: 各セクション間で矛盾がないか確認
   - 機能要件と受け入れ基準が対応しているか
   - 非機能要件と制約事項が矛盾していないか

3. **完全性**: Issue本文の情報を漏れなく反映
   - TODOセクションの項目をすべて機能要件に反映
   - 概要セクションの背景・目的を明確に記述

4. **検証可能性**: 各要件が検証可能（テスト可能）な形で記述
   - 受け入れ基準は Given-When-Then 形式で記述
   - 成功条件・失敗条件が明確に識別可能

5. **レビューされることを意識**: 作成した要件定義書はクリティカルシンキングレビューが実施されます
   - 品質ゲート（4つの必須要件）を最優先で満たすこと
   - ブロッカー（次フェーズに進めない問題）がないように注意
   - 改善提案は歓迎されるが、ブロッカーは修正が必須
