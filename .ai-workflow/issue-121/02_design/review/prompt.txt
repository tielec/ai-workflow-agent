### 本文

## 概要

AI Workflow Agentにリポジトリを探索して自動的にIssueを作成する機能を追加する。現在はIssueを手動で作成してからワークフローを実行しているが、Issue作成自体もAIエージェントに任せることで、開発効率をさらに向上させる。

## 背景・動機

- Issue作成作業が負担になってきた
- リポジトリ全体を探索して潜在的な問題や改善点を発見したい
- 既存のIssueと重複しないように自動的にチェックしたい

## 実装すべき機能

### 1. リポジトリ探索エンジン

コードベース全体を探索し、以下の3つのカテゴリで問題・改善点を検出する：

#### a. 潜在的な不具合の報告
- エラーハンドリングの欠如
- 型安全性の問題（`any`型の過剰使用等）
- リソースリーク（unclosed streams, event listeners等）
- セキュリティ上の懸念（credentials漏洩、XSS脆弱性等）
- 依存関係の脆弱性

#### b. リファクタリングすべきコードやドキュメントの報告
- 大きすぎるファイル・関数（行数基準）
- 重複コード
- 複雑度が高いコード（Cyclomatic Complexity）
- 未使用のインポート・変数
- ドキュメントの欠落または古い記述

#### c. 将来的な機能拡張のアイデアの提案

**既存機能の改善**:
- ユーザビリティ改善の提案
- パフォーマンス最適化の提案
- テストカバレッジ向上の提案
- CI/CD改善の提案

**創造的な新機能の提案**（柔軟な発想で）:
- **他ツール・サービスとの連携アイデア**
  - Slack/Discord通知、Jiraチケット自動生成、VSCode拡張機能等
- **ワークフロー自動化の拡張**
  - 定期実行による技術的負債の可視化、依存関係自動更新等
- **開発者体験（DX）向上**
  - インタラクティブCLI、進捗ダッシュボード、AIペアプログラミング支援等
- **品質保証の強化**
  - セキュリティスキャン統合、アクセシビリティチェック、負荷テスト自動化等
- **データ駆動の意思決定支援**
  - メトリクスダッシュボード、コスト分析レポート、開発速度予測等
- **エコシステム拡張**
  - プラグインシステム、カスタムフェーズ定義、テンプレート共有マーケットプレイス等
- **その他、リポジトリ内容から着想を得た独自のアイデア**

AIエージェントには、既存の枠にとらわれず、リポジトリの特性やコミット履歴、ドキュメント内容から「こういう機能があったらもっと便利になる」という提案を自由に発想してほしい。

### 2. 重複Issue検出機能

- GitHub APIを使用して既存のIssue一覧を取得
- LLM（OpenAI APIまたはAnthropic API）を使用して意味的類似度を判定
- 重複と判定されたIssueは作成をスキップし、ログに記録

### 3. Issue自動作成フロー

```bash
# 新しいCLIコマンド
node dist/index.js auto-issue [OPTIONS]

# オプション例
--category <bug|refactor|enhancement|all>  # 作成するIssueのカテゴリ
--limit <NUM>                              # 作成するIssueの最大数（デフォルト: 5）
--dry-run                                  # 実際には作成せず、候補のみ表示
--similarity-threshold <0-1>               # 重複判定の類似度閾値（デフォルト: 0.8）
--creative-mode                            # 創造的な機能提案を優先的に生成
```

### 4. Issue作成テンプレート

生成されるIssueは以下の形式に従う：

```markdown
## 概要
[AIが生成した問題・改善点のサマリー]

## 詳細
[具体的な問題の説明]

## 該当箇所
- ファイル: src/path/to/file.ts:123-145
- 関連コード: [コードスニペット]

## 提案される解決策
[AIによる解決策の提案]

## 期待される効果
[この改善による具体的なメリット]

## 優先度
[Low / Medium / High]

## カテゴリ
[bug / refactor / enhancement]

```

## 技術的な実装アプローチ

### 1. 新しいコマンドハンドラ
- `src/commands/auto-issue.ts`: `handleAutoIssueCommand()`

### 2. リポジトリ探索エンジン
- `src/core/repository-analyzer.ts`: コードベース解析
  - 静的解析（TypeScript AST解析）
  - パターンマッチング（ESLintルール活用）
  - メトリクス計測（複雑度、行数等）
  - コミット履歴分析（頻繁に変更されるファイル、バグ修正パターン等）
  - ドキュメント分析（README、CLAUDE.md、ARCHITECTURE.md等）

### 3. 重複検出エンジン
- `src/core/issue-deduplicator.ts`: 既存Issue比較
  - GitHub API連携（既存Issue取得）
  - LLM活用（意味的類似度判定）
  - 類似度スコアリング

### 4. Issue生成エンジン
- `src/core/issue-generator.ts`: Issue自動作成
  - テンプレート生成
  - LLMによる本文生成（創造的な提案を引き出すプロンプト設計）
  - GitHub API経由でIssue作成

### 5. 創造的提案のためのLLMプロンプト設計
- リポジトリの特性を分析し、文脈に応じた提案を生成
- 既存機能の組み合わせや拡張による新しいユースケースの発見
- 他のプロジェクトやツールからのベストプラクティス適用

### 6. 既存モジュールの活用
- `src/core/github-client.ts`: GitHub API連携（Issue作成・取得）
- `src/core/config.ts`: 環境変数管理
- `src/utils/logger.ts`: ロギング

## 期待される効果

1. **開発効率の向上**: Issue作成作業の自動化
2. **品質向上**: 見落としがちな潜在的問題の早期発見
3. **継続的改善**: 定期的な自動Issue作成による技術的負債の可視化
4. **重複削減**: 既存Issueとの重複を自動的に回避
5. **イノベーション促進**: AIによる創造的な機能提案により、新しい可能性を発見

## 代替案・考慮事項

### 代替案1: 完全自動実行（Cron Job）
- 週次で自動的にリポジトリを探索し、Issueを作成
- GitHub Actionsとの連携

### 代替案2: 段階的実装
- Phase 1: バグ検出のみ実装
- Phase 2: リファクタリング検出を追加
- Phase 3: 機能拡張提案を追加（創造的提案を含む）

### 考慮事項
- **LLMコスト**: 大規模リポジトリでのトークン使用量
- **誤検知**: AIによる誤ったIssue作成の可能性
- **Issue spam**: あまりに多くのIssueが作成されることを防ぐ制限
- **プライバシー**: プライベートリポジトリのコード内容をLLMに送信することのセキュリティリスク
- **提案品質の担保**: 創造的提案が実際に有用かどうかの判定基準

## レビュー姿勢

このレビューは「**80点で十分**」の原則に基づいて実施してください：

1. **完璧ではなく、十分を目指す**
   - 設計書が次フェーズ（テストシナリオ作成）に進める状態であれば合格
   - 細かい表現の改善は改善提案として記載（ブロッカーにしない）

2. **ブロッカーと改善提案を明確に区別**
   - ブロッカー: 次フェーズ（テストシナリオ）に進めない重大な問題
   - 改善提案: 次フェーズに進めるが、改善が望ましい事項

3. **実用的でバランスの取れた判断**
   - プロジェクトを前に進めることを最優先
   - 実装フェーズで対応可能な問題はブロッカーにしない

4. **建設的なフィードバック**
   - 「ダメ」ではなく「こうすればより良い」という表現
   - 具体的な改善案を提示

## レビュー観点

### 1. 戦略判断の妥当性（最重要）

**実装戦略（CREATE/EXTEND/REFACTOR）**:
- 判断根拠が具体的かつ論理的に記載されているか
- 要件定義書の内容と整合しているか
- 既存コードへの影響が適切に評価されているか

**テスト戦略（UNIT_ONLY/INTEGRATION_ONLY/BDD_ONLY/UNIT_INTEGRATION/UNIT_BDD/INTEGRATION_BDD/ALL）**:
- 判断根拠が具体的かつ論理的に記載されているか
- 機能の複雑度に見合ったテスト戦略か
- 要件定義の受け入れ基準と整合しているか

**テストコード戦略（EXTEND_TEST/CREATE_TEST/BOTH_TEST）**:
- 判断根拠が具体的かつ論理的に記載されているか
- 既存テストとの関係が明確か

### 2. 影響範囲分析の適切性

- 既存コードへの影響が網羅的に分析されているか
- 依存関係が正しく把握されているか
- マイグレーションの必要性が評価されているか

### 3. ファイルリストの完全性

- 新規作成ファイルがリストアップされているか
- 修正が必要な既存ファイルがリストアップされているか
- パスが具体的で実装可能か

### 4. 設計の実装可能性

- 設計が具体的で実装者が迷わないか
- 技術的に実装可能な設計か
- 既存プロジェクトの規約・パターンに準拠しているか

### 5. 要件との対応（トレーサビリティ）

- 要件定義書の各要件に対応する設計があるか
- 要件の漏れがないか

### 6. セキュリティ考慮

- セキュリティリスクが識別されているか
- 対策が具体的か

### 7. 非機能要件への対応

- パフォーマンス考慮があるか
- スケーラビリティ考慮があるか
- 保守性考慮があるか

## ⚠️ 最重要: 品質ゲートは絶対条件

**品質ゲート（5項目）は必須要件です。1つでも満たされていない場合、判定は自動的にFAILになります。**

- 品質ゲートは「最低限クリアすべき基準」であり、交渉の余地はありません
- 「80点で十分」の原則は、品質ゲートを満たした上での改善提案に適用されます
- 品質ゲートを満たさない状態で次フェーズに進むことは許されません

## ブロッカー（BLOCKER）と改善提案（SUGGESTION）の区別

### ブロッカー（BLOCKER）: 次フェーズに進めない重大な問題

**ブロッカーの例**:
- 3つの戦略判断（実装・テスト・テストコード）のいずれかが欠落
- 判断根拠が記載されていない、または論理的に破綻している
- 変更ファイルリストが未記載
- 設計が実装不可能（技術的制約に違反）
- 既存システムを破壊する設計
- 重大なセキュリティリスクが未対策
- 要件との対応が不明確

### 改善提案（SUGGESTION）: 次フェーズに進めるが、改善が望ましい事項

**改善提案の例**:
- より良い設計パターンの提案
- パフォーマンス最適化の余地
- ドキュメントの充実（例: 図表追加）
- エッジケースの追加考慮
- より詳細な影響範囲分析
- セキュリティ対策の強化

### 判定ルール

#### PASS（合格）

以下の**すべて**を満たす場合のみPASS：
- ✅ **5つの品質ゲートすべてが満たされている**（1つでも×があればFAIL）
- ✅ ブロッカーが存在しない
- ✅ 軽微な改善提案のみ（または改善提案なし）

#### PASS_WITH_SUGGESTIONS（条件付き合格）

以下の**すべて**を満たす場合のみPASS_WITH_SUGGESTIONS：
- ✅ **5つの品質ゲートすべてが満たされている**（1つでも×があればFAIL）
- ✅ ブロッカーは存在しない
- ✅ 次フェーズに進めるが、検討すべき改善点がある

#### FAIL（不合格）

以下の**いずれか1つでも**該当すればFAIL：
- ❌ **品質ゲート（5項目）のうち1つでも満たされていない**
- ❌ ブロッカーが1つ以上存在する

**重要: 品質ゲートは絶対条件であり、1つでも満たされていなければ自動的にFAILです。**
