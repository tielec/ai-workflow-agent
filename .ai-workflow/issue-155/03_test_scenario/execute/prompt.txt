## 概要

`src/core/repository-analyzer.ts`内の`analyze()`メソッドと`analyzeForRefactoring()`メソッドに約150行の重複したエージェントフォールバックロジックが存在します。この重複を解消することで、保守性の向上とエージェント実行ロジックの一元管理を実現します。

## 詳細

**現状の問題点**

- `analyze()`メソッド（234-305行）と`analyzeForRefactoring()`メソッド（315-386行）が、各71行の同一パターンのコードを含んでいます
- 両メソッドともテンプレート読込→出力ファイル生成→変数置換→エージェント実行→ファイル読取→バリデーション→クリーンアップという同じ処理フローを実装しています
- 特に258-281行と338-361行のエージェント選択ロジックが完全に重複しています

**技術的負債の影響**

- エージェントフォールバック動作の変更が必要な場合、2箇所以上の修正が必要となり、修正漏れのリスクが高まります
- 約150行の重複コードがメンテナンスコストを増加させ、バグ混入の可能性を高めています
- コードの可読性が低下し、新規参加者の理解を妨げています

**リファクタリングの具体的な内容**

- `executeAgentWithFallback(prompt: string, outputFile: string, candidateType: 'bug'|'refactor')`という共通メソッドを抽出します
- テンプレートベースの実行パターンを一般化し、候補タイプに応じた動作の切り替えを実現します
- バリデーションロジックも単一メソッドに統合します

**期待される改善効果**

- 約150行のコード削減による保守性の大幅な向上
- エージェント実行ロジックの変更を1箇所で管理可能
- テストカバレッジの向上（共通ロジックのテストに集中できる）
- コードの可読性と理解容易性の向上

## 影響範囲

**影響を受けるモジュール/ファイル**

- `src/core/repository-analyzer.ts`（主要な変更対象）
- `analyze()`メソッドを呼び出している全てのモジュール（インターフェースは維持）
- `analyzeForRefactoring()`メソッドを呼び出している全てのモジュール（インターフェースは維持）

**依存関係への影響**

- 外部APIは変更しないため、呼び出し側への影響はありません
- 内部実装の変更のみで、既存の依存関係は維持されます

**テストへの影響**

- 既存のユニットテストは基本的に影響を受けません（パブリックインターフェース不変）
- 新規に抽出される`executeAgentWithFallback()`メソッド用のテストケース追加が推奨されます
- 統合テストは既存のものをそのまま利用可能です

**破壊的変更の有無**

- 破壊的変更はありません
- パブリックメソッドのシグネチャと振る舞いは完全に維持されます

## 推奨される改善策

**推奨するリファクタリング手法**

1. **Extract Method パターン**を適用し、共通ロジックを`executeAgentWithFallback()`として抽出
2. **Template Method パターン**を活用し、候補タイプ別の振る舞いの差異をパラメータ化
3. バリデーションロジックを`validateAnalysisResult(result: any, candidateType: 'bug'|'refactor'): boolean`として独立させる

**段階的な実施計画**

1. **Phase 1**: 既存のテストケースを確認し、リグレッション検証の準備を行う
2. **Phase 2**: `executeAgentWithFallback()`メソッドを実装し、ユニットテストを作成
3. **Phase 3**: `analyze()`メソッドを新しい共通メソッドを使用する形にリファクタリング
4. **Phase 4**: `analyzeForRefactoring()`メソッドを新しい共通メソッドを使用する形にリファクタリング
5. **Phase 5**: 全テストを実行し、動作の一貫性を確認

**参考となるパターンやベストプラクティス**

- Martin Fowler『リファクタリング』の「Extract Method」「Template Method」パターン
- DRY原則（Don't Repeat Yourself）の徹底
- 単一責任の原則（SRP）に基づくメソッド分割

## テスト戦略別の対応

Phase 2で決定されたテスト戦略に応じて、以下のテストシナリオを作成してください：

### UNIT_ONLY: Unitテストのみ
- 各関数・メソッド単位のテストケース
- 正常系・異常系・境界値テスト
- モック/スタブを使用した単体テスト

### INTEGRATION_ONLY: Integrationテストのみ
- コンポーネント間の連携テスト
- 外部システムとの統合テスト

### BDD_ONLY: BDDテストのみ
- ユーザーストーリーベースのシナリオ
- Given-When-Then形式
- ビジネス要件との対応

### UNIT_INTEGRATION: Unit + Integration

- BDDシナリオ（上記）

- Unitテストシナリオ（上記）

## テストシナリオの構成

### 1. テスト戦略サマリー
- 選択されたテスト戦略（Phase 2から引用）
- テスト対象の範囲
- テストの目的

### 2. Unitテストシナリオ（該当する場合）

各関数・メソッドについて：

**テストケース名**: （関数名_正常系/異常系/境界値）

- **目的**: このテストで検証すること
- **前提条件**: テスト実行前の状態
- **入力**: 関数への入力パラメータ
- **期待結果**: 期待される出力・状態変化
- **テストデータ**: 使用するテストデータ

**例**:
```
テストケース名: calculate_total_正常系
目的: 合計金額が正しく計算されることを検証
前提条件: 商品リストが存在する
入力: items = [{price: 100, quantity: 2}, {price: 200, quantity: 1}]
期待結果: 400が返される
テストデータ: 上記items
```

### 3. Integrationテストシナリオ（該当する場合）

各統合ポイントについて：

**シナリオ名**: （統合する2つのコンポーネント名）

- **目的**: この統合で検証すること
- **前提条件**: 統合テスト実行前の状態
- **テスト手順**: ステップバイステップの手順
- **期待結果**: 統合後の期待される動作
- **確認項目**: 確認すべきポイントのチェックリスト

### 4. BDDシナリオ（該当する場合）

ユーザーストーリーごとに：

**Feature**: （機能名）

**Scenario**: （シナリオ名）

- **Given** (前提条件): システムの初期状態
- **When** (操作): ユーザーが実行するアクション
- **Then** (結果): 期待される結果

**例**:
```gherkin
Feature: ユーザー登録機能

Scenario: 新規ユーザーが正常に登録できる
  Given ユーザーがログインしていない
  When ユーザーが有効なメールアドレスとパスワードで登録フォームを送信する
  Then ユーザーアカウントが作成される
  And 確認メールが送信される
  And ユーザーがログイン画面にリダイレクトされる
```

### 5. テストデータ
- 各テストシナリオで使用するテストデータ
- 正常データ、異常データ、境界値データ

### 6. テスト環境要件
- 必要なテスト環境（ローカル、CI/CD等）
- 必要な外部サービス・データベース
- モック/スタブの必要性

## 品質ゲート（Phase 3）

テストシナリオは以下の品質ゲートを満たす必要があります：

- [ ] **Phase 2の戦略に沿ったテストシナリオである**
- [ ] **主要な正常系がカバーされている**
- [ ] **主要な異常系がカバーされている**
- [ ] **期待結果が明確である**

これらの品質ゲートは**必須要件**です。作成後、クリティカルシンキングレビューが実施され、品質ゲートを満たさない場合は修正が必要になります。

## 注意事項

1. **戦略準拠**: Phase 2のテスト戦略を厳守してください
   - 戦略に含まれないテスト種別は作成不要
   - 例: UNIT_ONLYの場合、Integration/BDDシナリオは不要

2. **要件との対応**: 要件定義書の各要件がテストでカバーされていることを確認
   - 受け入れ基準がテストシナリオに反映されているか
   - 機能要件がすべてテスト対象に含まれているか

3. **実行可能性**: 実際に実行可能なテストシナリオを記述
   - 曖昧な表現を避ける
   - 具体的な入力・出力を記載
   - 検証可能な期待結果

4. **優先度**: すべてのエッジケースではなく、主要なケースに注目
   - 80点で十分（完璧を目指さない）
   - クリティカルパス優先
   - 高リスク領域を優先的にカバー

5. **レビューされることを意識**: テストシナリオはクリティカルシンキングレビューが実施されます
   - 品質ゲート（4つの必須要件）を最優先で満たすこと
   - ブロッカー（次フェーズに進めない問題）がないように注意
