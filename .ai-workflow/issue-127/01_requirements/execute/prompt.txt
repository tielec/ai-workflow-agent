## 実装内容

### 1. リファクタリング検出機能の追加
`src/core/repository-analyzer.ts` を拡張:

- **Claude Code / Codex エージェントを使用**してリファクタリング候補を検出
- 既存の `--agent` オプションを継承（auto/codex/claude）
- **言語非依存**: TypeScript以外のリポジトリ（Python, Go, Java等）にも対応

#### 検出パターン（エージェントに指示）

##### a. コード品質問題
- 大きすぎるファイル・関数
- 複雑すぎる条件分岐・ネスト
- 責務が多すぎるクラス・モジュール

##### b. コード重複
- 類似したコードブロック
- コピー＆ペーストされたロジック
- 共通化可能なパターン

##### c. 未使用コード
- 未使用のインポート・変数・関数
- Dead code
- 到達不能コード

##### d. ドキュメント品質
- コメント・ドキュメントの欠落
- 古い・不正確なドキュメント
- README等の更新漏れ

### 2. エージェントプロンプト設計
リファクタリング検出用のプロンプトテンプレート:

```
このリポジトリをレビューし、リファクタリングが必要な箇所を特定してください。

以下の観点で分析してください：
1. コード品質: 大きすぎるファイル・関数、複雑な条件分岐
2. コード重複: 共通化可能なパターン
3. 未使用コード: 不要なインポート、Dead code
4. ドキュメント: 欠落・古い記述

各候補について以下の形式で報告してください：
- ファイルパス
- 行範囲（該当する場合）
- 問題の種類
- 具体的な問題点
- 推奨される改善策
```

### 3. Issue生成テンプレートの拡張
- リファクタリング用のテンプレート追加
- 具体的なリファクタリング手順の提案
- 期待される効果（可読性向上、保守性向上等）

### 4. カテゴリフィルタリング
- `--category refactor` オプションの実装
- `--category all` でバグ + リファクタリングを同時検出

## 技術的詳細

### エージェント活用方針

| 機能 | 実装方法 | 理由 |
|------|----------|------|
| コード探索・リファクタ検出 | Claude Code / Codex | ファイル読み込み、コード構造理解が必要 |
| Issue本文生成 | Claude Code / Codex | 文脈を踏まえた改善提案の生成 |
| 重複検出（類似度判定） | OpenAI API | Phase 1で実装済みの機能を再利用 |

### 検出結果の構造
```typescript
interface RefactorCandidate {
  type: 'large-file' | 'large-function' | 'high-complexity' | 'duplication' | 'unused-code' | 'missing-docs';
  filePath: string;
  lineRange?: { start: number; end: number };
  description: string;
  suggestion: string;
  priority: 'low' | 'medium' | 'high';
}
```

### エージェント実行フロー
```
1. handleAutoIssueCommand(category: 'refactor')
   │
2. RepositoryAnalyzer.analyzeForRefactoring()
   ├── エージェントにリファクタリング候補探索を指示
   ├── エージェント出力をパース
   └── RefactorCandidate[] を返却
   │
3. IssueDeduplicator.filterDuplicates()
   └── Phase 1の機能を再利用
   │
4. IssueGenerator.generateRefactorIssue()
   ├── エージェントにIssue本文生成を指示
   └── GitHub API でIssue作成
```

## 要件定義書の構成

以下のセクションを含む要件定義書を作成してください：

### 0. Planning Documentの確認（Planning Phaseが実行されている場合）
- 開発計画の全体像を把握
- スコープ、技術選定、リスク、スケジュールを確認
- Planning Documentで策定された戦略を踏まえて要件定義を実施

### 1. 概要
- Issue本文の「## 概要」セクションを要約
- 背景と目的を明確に記述
- ビジネス価値・技術的価値を説明

### 2. 機能要件
- Issue本文の「## TODO」セクションから機能要件を抽出
- 各要件を明確かつ検証可能な形で記述
- 優先度（高/中/低）を付与

### 3. 非機能要件
- パフォーマンス要件
- セキュリティ要件
- 可用性・信頼性要件
- 保守性・拡張性要件

### 4. 制約事項
- 技術的制約（使用技術、既存システムとの整合性）
- リソース制約（時間、人員、予算）
- ポリシー制約（セキュリティポリシー、コーディング規約）

### 5. 前提条件
- システム環境
- 依存コンポーネント
- 外部システム連携

### 6. 受け入れ基準
- 各機能要件の受け入れ基準（Given-When-Then形式推奨）
- テスト可能な形で記述

### 7. スコープ外
- 明確にスコープ外とする事項
- 将来的な拡張候補

## 品質ゲート（Phase 1）

作成する要件定義書は、以下の品質ゲートを満たす必要があります：

- [ ] **機能要件が明確に記載されている**
- [ ] **受け入れ基準が定義されている**
- [ ] **スコープが明確である**
- [ ] **論理的な矛盾がない**

これらの品質ゲートは**必須要件**です。作成後、クリティカルシンキングレビューが実施され、品質ゲートを満たさない場合は修正が必要になります。

## 注意事項

1. **具体性**: 曖昧な表現を避け、具体的かつ測定可能な要件を記述
   - NG例: "適切に処理する"、"必要に応じて対応"
   - OK例: "3秒以内にレスポンスを返す"、"エラー発生時は管理者に通知"

2. **整合性**: 各セクション間で矛盾がないか確認
   - 機能要件と受け入れ基準が対応しているか
   - 非機能要件と制約事項が矛盾していないか

3. **完全性**: Issue本文の情報を漏れなく反映
   - TODOセクションの項目をすべて機能要件に反映
   - 概要セクションの背景・目的を明確に記述

4. **検証可能性**: 各要件が検証可能（テスト可能）な形で記述
   - 受け入れ基準は Given-When-Then 形式で記述
   - 成功条件・失敗条件が明確に識別可能

5. **レビューされることを意識**: 作成した要件定義書はクリティカルシンキングレビューが実施されます
   - 品質ゲート（4つの必須要件）を最優先で満たすこと
   - ブロッカー（次フェーズに進めない問題）がないように注意
   - 改善提案は歓迎されるが、ブロッカーは修正が必須
