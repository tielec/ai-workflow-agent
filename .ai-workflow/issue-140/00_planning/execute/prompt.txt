### 本文

## 概要
`src/core/claude-agent-client.ts`の`fillTemplate`メソッドにおいて、ユーザー入力を動的にRegExpオブジェクトに埋め込んでいるため、ReDoS（Regular Expression Denial of Service）攻撃に対する脆弱性が存在します。悪意のある特殊正規表現文字を含むテンプレート変数キーが渡された場合、CPUリソースを大量に消費させることが可能です。

## 詳細
**問題が発生する条件**
- `fillTemplate`メソッド（88行目付近）において、テンプレート変数のキーを`new RegExp()`コンストラクタに直接渡してパターンマッチを行っている
- ユーザーが制御可能なテンプレート変数のキー名に、正規表現の特殊文字（`.*+?^${}()|[]\`など）が含まれる場合

**現在の動作**
- テンプレート変数キーをエスケープせずにRegExpオブジェクトに渡しているため、特殊文字が正規表現メタ文字として解釈される
- 悪意のある入力（例：`(a+)+b`のようなバックトラッキングを引き起こすパターン）により、指数関数的な計算量が発生し、アプリケーションが応答不能になる可能性がある

**期待される動作**
- テンプレート変数キーは文字列リテラルとして扱われるべき
- 特殊文字が含まれていても安全に処理され、パフォーマンスへの影響がない

**根本原因**
- 動的なRegExp生成時に入力値のサニタイズ・エスケープ処理が行われていない
- セキュリティを考慮しない文字列置換の実装パターンが採用されている

## 影響範囲
**影響を受ける機能**
- `fillTemplate`メソッドを使用するすべてのテンプレート処理機能
- Claude Agent SDKのクライアント初期化およびテンプレート変数を扱うすべてのワークフロー

**影響を受けるユーザー**
- Claude Agent SDKを利用するすべての開発者および最終ユーザー
- 特に、外部からのテンプレート変数名を受け入れるアプリケーション

**セキュリティへの影響**
- DoS攻撃のリスク（重大度：High）
- サービスの可用性が損なわれる可能性
- 認証なしで攻撃可能な場合、深刻な脅威となる

**パフォーマンスへの影響**
- 悪意のある入力により、CPUリソースが枯渇し、アプリケーション全体が停止する可能性
- 同一プロセス内の他の処理にも影響を及ぼす

## 修正案
**修正すべきファイルとコード箇所**
- ファイル：`src/core/claude-agent-client.ts`
- 対象行：88行目付近の`fillTemplate`メソッド内のRegExp生成コード

**修正内容の説明**
以下のいずれかの方法で修正することを推奨します：

1. **String.prototype.replaceAll()を使用する方法（推奨）**
   - RegExpを使わず、文字列の`replaceAll()`メソッドを使用して安全に置換
   - 例：`template.replaceAll(key, value)`

2. **正規表現エスケープライブラリの導入**
   - `escape-string-regexp`などのライブラリを使用してキーをエスケープ
   - 例：`const escapedKey = key.replace(/[.*+?^${}()|[\]\\]/g, '\\{issue_info}'); const regex = new RegExp(escapedKey, 'g');`

3. **手動エスケープ処理の実装**
   - 正規表現の特殊文字を手動でエスケープする関数を実装
   - エスケープ対象：`. * + ? ^ $ { } ( ) | [ ] \`

**テスト方法の提案**
- 特殊文字を含むテンプレート変数キーでのユニットテスト追加（例：`{{a+b}}`, `{{(test)*}}`, `{{.*}}`など）
- ReDoSパターンでのパフォーマンステスト（タイムアウト検証）
- 既存のテンプレート処理が正常に動作することの回帰テスト

## プロジェクト計画書の構成

以下のセクションを含むプロジェクト計画書を作成してください：

### 1. Issue分析
- **複雑度**: 簡単/中程度/複雑を判定
  - 簡単: 単一ファイルの修正、シンプルな機能追加（見積もり: ~4時間）
  - 中程度: 複数ファイルの修正、既存機能の拡張（見積もり: 4~16時間）
  - 複雑: アーキテクチャ変更、新規サブシステム追加（見積もり: 16時間~）
- **見積もり工数**: X時間（根拠を記載）
- **リスク評価**: 低/中/高

### 2. 実装戦略判断

**重要**: このセクションは最も重要です。Phase 2（Design）で使用されるため、必ず明記してください。

#### 実装戦略: CREATE / EXTEND / REFACTOR

**判断根拠**:
- CREATE: 新規ファイル・クラス・モジュールの作成が中心
- EXTEND: 既存コードの拡張（機能追加）が中心
- REFACTOR: 既存コードのリファクタリング（構造改善）が中心

（判断理由を具体的に記載）

#### テスト戦略: UNIT_ONLY / INTEGRATION_ONLY / BDD_ONLY / UNIT_INTEGRATION / UNIT_BDD / INTEGRATION_BDD / ALL

**判断根拠**:
- UNIT_ONLY: ユニットテストのみ（単純なロジック、外部依存なし）
- INTEGRATION_ONLY: インテグレーションテストのみ（外部システム連携中心）
- BDD_ONLY: BDDテストのみ（ユーザーストーリー中心）
- UNIT_INTEGRATION: ユニット + インテグレーション（一般的な機能開発）
- UNIT_BDD: ユニット + BDD（エンドユーザー向け機能）
- INTEGRATION_BDD: インテグレーション + BDD（システム間連携＋ユーザーストーリー）
- ALL: すべてのテストレベル（大規模な変更、高リスク）

（判断理由を具体的に記載）

#### テストコード戦略: EXTEND_TEST / CREATE_TEST / BOTH_TEST

**判断根拠**:
- EXTEND_TEST: 既存テストファイルに追加（既存機能の拡張時）
- CREATE_TEST: 新規テストファイル作成（新規機能追加時）
- BOTH_TEST: 両方（既存テスト拡張 + 新規テスト作成）

（判断理由を具体的に記載）

これらの品質ゲートは**必須要件**です。作成後、クリティカルシンキングレビューが実施され、品質ゲートを満たさない場合は修正が必要になります。

## 注意事項

1. **実装戦略の判断**: Phase 2で使用されるため、必ず明記してください
   - 実装戦略: CREATE/EXTEND/REFACTOR
   - テスト戦略: UNIT_ONLY/.../ALL
   - テストコード戦略: EXTEND_TEST/CREATE_TEST/BOTH_TEST

2. **具体性**: 曖昧な表現を避け、具体的かつ測定可能な計画を記述
   - NG例: "適切に対応する"、"必要に応じて実施"
   - OK例: "Phase 2で詳細設計を実施"、"最大3回のリトライを実施"

3. **実現可能性**: 見積もりが現実的か、リソースは十分か確認
   - タスク粒度が大きすぎないか（1タスク = 1~4時間）
   - 依存関係が正確に記載されているか

4. **リスク分析の網羅性**: 重要なリスクが見逃されていないか確認
   - 技術的リスク、スコープリスク、リソースリスク、依存リスク

5. **レビューされることを意識**: 作成した計画書はクリティカルシンキングレビューが実施されます
   - 品質ゲート（6つの必須要件）を最優先で満たすこと
   - ブロッカー（次フェーズに進めない問題）がないように注意
   - 改善提案は歓迎されるが、ブロッカーは修正が必須
