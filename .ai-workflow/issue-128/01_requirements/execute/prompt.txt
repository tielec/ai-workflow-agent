## 実装内容

### 1. リポジトリ特性分析
`src/core/repository-analyzer.ts` を拡張:

- **Claude Code / Codex エージェントを使用**してリポジトリを深く分析
- 既存の `--agent` オプションを継承（auto/codex/claude）

#### 分析観点（エージェントに指示）

##### a. プロジェクト構造分析
- 使用技術スタック（言語、フレームワーク、ライブラリ）
- アーキテクチャパターン
- ディレクトリ構造とモジュール構成

##### b. 既存機能の理解
- 主要な機能・ワークフロー
- 設定可能なオプション
- 拡張ポイント

##### c. ドキュメント分析
- README.md、設計ドキュメント等の解析
- 既存のロードマップ・TODO項目
- 技術的負債の記述

### 2. 創造的提案生成
- **Claude Code / Codex エージェントを使用**して機能提案を生成
- リポジトリの特性を踏まえた文脈認識型の提案

#### 提案カテゴリ（エージェントに指示）

##### a. 既存機能の改善提案
- ユーザビリティ改善
- パフォーマンス最適化
- テストカバレッジ向上
- CI/CD改善

##### b. 新機能の提案
- 他ツール・サービスとの連携（Slack、Jira、VSCode等）
- ワークフロー自動化の拡張
- 開発者体験（DX）向上
- 品質保証の強化
- データ駆動の意思決定支援
- エコシステム拡張（プラグイン、テンプレート等）

### 3. エージェントプロンプト設計
創造的提案用のプロンプトテンプレート:

```
このリポジトリを深く分析し、機能拡張の提案を生成してください。

## リポジトリ分析
まず以下を把握してください：
- 技術スタック、アーキテクチャ
- 主要機能、ワークフロー

## 提案生成
以下の観点で「こういう機能があったらもっと便利になる」という提案をしてください：

1. 既存機能の改善
   - ユーザビリティ、パフォーマンス、テスト、CI/CD

2. 新機能の提案（創造的に）
   - 他ツール連携、自動化拡張、DX向上、品質保証、エコシステム拡張

既存の枠にとらわれず、リポジトリの特性から着想を得た独自のアイデアも歓迎します。

## 出力形式
各提案について：
- タイトル
- 概要（2-3文）
- 期待される効果
- 実装の難易度（small/medium/large）
- 関連するファイル・モジュール
```

## 技術的詳細

### エージェント活用方針

| 機能 | 実装方法 | 理由 |
|------|----------|------|
| リポジトリ分析 | Claude Code / Codex | ファイル読み込み、全体構造の理解が必要 |
| 創造的提案生成 | Claude Code / Codex | 文脈を踏まえた創造的発想が必要 |
| 重複検出 | OpenAI API | Phase 1の機能を再利用 |

### 提案の構造
```typescript
interface EnhancementProposal {
  type: 'improvement' | 'integration' | 'automation' | 'dx' | 'quality' | 'ecosystem';
  title: string;
  description: string;
  rationale: string;  // なぜこの提案が有用か
  implementation_hints: string[];
  expected_impact: 'low' | 'medium' | 'high';
  effort_estimate: 'small' | 'medium' | 'large';
  related_files: string[];
}
```

### エージェント実行フロー
```
1. handleAutoIssueCommand(category: 'enhancement')
   │
2. RepositoryAnalyzer.analyzeForEnhancements()
   ├── エージェントにリポジトリ分析を指示
   ├── エージェントに創造的提案生成を指示
   ├── エージェント出力をパース
   └── EnhancementProposal[] を返却
   │
3. IssueDeduplicator.filterDuplicates()
   └── Phase 1の機能を再利用
   │
4. IssueGenerator.generateEnhancementIssue()
   ├── 提案をIssue形式に変換
   └── GitHub API でIssue作成
```

## 要件定義書の構成

以下のセクションを含む要件定義書を作成してください：

### 0. Planning Documentの確認（Planning Phaseが実行されている場合）
- 開発計画の全体像を把握
- スコープ、技術選定、リスク、スケジュールを確認
- Planning Documentで策定された戦略を踏まえて要件定義を実施

### 1. 概要
- Issue本文の「## 概要」セクションを要約
- 背景と目的を明確に記述
- ビジネス価値・技術的価値を説明

### 2. 機能要件
- Issue本文の「## TODO」セクションから機能要件を抽出
- 各要件を明確かつ検証可能な形で記述
- 優先度（高/中/低）を付与

### 3. 非機能要件
- パフォーマンス要件
- セキュリティ要件
- 可用性・信頼性要件
- 保守性・拡張性要件

### 4. 制約事項
- 技術的制約（使用技術、既存システムとの整合性）
- リソース制約（時間、人員、予算）
- ポリシー制約（セキュリティポリシー、コーディング規約）

### 5. 前提条件
- システム環境
- 依存コンポーネント
- 外部システム連携

### 6. 受け入れ基準
- 各機能要件の受け入れ基準（Given-When-Then形式推奨）
- テスト可能な形で記述

### 7. スコープ外
- 明確にスコープ外とする事項
- 将来的な拡張候補

## 品質ゲート（Phase 1）

作成する要件定義書は、以下の品質ゲートを満たす必要があります：

- [ ] **機能要件が明確に記載されている**
- [ ] **受け入れ基準が定義されている**
- [ ] **スコープが明確である**
- [ ] **論理的な矛盾がない**

これらの品質ゲートは**必須要件**です。作成後、クリティカルシンキングレビューが実施され、品質ゲートを満たさない場合は修正が必要になります。

## 注意事項

1. **具体性**: 曖昧な表現を避け、具体的かつ測定可能な要件を記述
   - NG例: "適切に処理する"、"必要に応じて対応"
   - OK例: "3秒以内にレスポンスを返す"、"エラー発生時は管理者に通知"

2. **整合性**: 各セクション間で矛盾がないか確認
   - 機能要件と受け入れ基準が対応しているか
   - 非機能要件と制約事項が矛盾していないか

3. **完全性**: Issue本文の情報を漏れなく反映
   - TODOセクションの項目をすべて機能要件に反映
   - 概要セクションの背景・目的を明確に記述

4. **検証可能性**: 各要件が検証可能（テスト可能）な形で記述
   - 受け入れ基準は Given-When-Then 形式で記述
   - 成功条件・失敗条件が明確に識別可能

5. **レビューされることを意識**: 作成した要件定義書はクリティカルシンキングレビューが実施されます
   - 品質ゲート（4つの必須要件）を最優先で満たすこと
   - ブロッカー（次フェーズに進めない問題）がないように注意
   - 改善提案は歓迎されるが、ブロッカーは修正が必須
